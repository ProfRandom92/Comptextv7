## Purpose

This walkthrough gives reviewers a no-local-execution path for inspecting the CompText V7 evidence chain. It focuses on the CI artifact narrative, explaining how CompText V7 evidence is produced, validated, and interpreted.

## What this demo proves
- **Deterministic state extraction**: The system can extract structured operational fields into a compact KVTC frame.
- **Fixture-bound reproducibility**: With the checked-in fixtures, replay and validation yield identical outcomes on every run without external calls.
- **Replay consistency**: For structured agent trace fixtures, the current validator confirms near-lossless replay of active tasks and recovery actions.
- **Auditability**: CI artifacts serve as inspectable snapshots, making any field-loss (operational drift) immediately visible in PRs.

## What this demo does not prove
- **Universal memory correctness**: The metrics are fixture-bound.
- **Autonomous execution**: Replay artifacts are deterministic snapshots, not live autonomous execution.
- **Zero semantic drift on unstructured text**: Dense paper replay remains lossy under current compression.
- **Long-horizon perfection**: Long-horizon stress tests show degradation in fidelity over time.
- **Production readiness**: This is a conservative research prototype, not a solved AI memory or clinical-grade workflow orchestrator.

## Step 1: inspect fixture inputs
1. Open the repository's `tests/fixtures/` or the source datasets used by the benchmarks.
2. Note that these are fixed, synthetic inputs representing agent traces (structured) and technical papers (dense prose).

## Step 2: inspect committed benchmark artifacts
1. Review `artifacts/agent_trace_replay_results.json` and `artifacts/paper_replay_results.json`.
2. Notice the metric values: you will see that structured agent traces replay near-losslessly under the current validator, while dense paper replay remains lossy.
3. Understand that these artifacts act as the static baseline for CI regressions.

## Step 3: inspect ReplayArtifactWriter output
1. Review the output generated by `ReplayArtifactWriter v1-alpha.1` (which may be found in CI logs or local validation runs under `artifacts/`).
2. Observe how the writer binds the operational state to a deterministic `artifactHash`.
3. Recognize that this hashing provides deterministic integrity for auditability, not cryptographic security.
4. Verify that no LLM judges, embeddings, vector DBs, or external APIs are used to produce or validate this artifact.

## Step 4: run validation commands
While this walkthrough focuses on cloud review, the evidence is generated via these commands:
```bash
npm run check
```
This single command ensures the repository layout, typechecks, static validation, builds, and deterministic Python validation tests all pass using the static artifacts.

## Step 5: interpret metrics safely
When reviewing PR outputs or benchmark reports, keep these conservative boundaries in mind:
- **Metrics are fixture-bound:** They prove behavior on these specific traces, not on unseen real-world data.
- **Operational drift:** A non-zero drift implies dropped structural fields, which is expected on dense text but minimized on structured logs.
- **No subjective judging:** The metrics are based on deterministic field matching, not AI-based grading.
